# Build & Deploy — Adams Industries Hugo Site

This skill builds the Hugo site and pushes changes to GitHub. Run it after every site change.

## Architecture

- **Site root (file tools):** `/Users/gatorboots/Desktop/adamsIndustries-www`
- **Site root (bash sandbox):** `/sessions/kind-dazzling-fermi/mnt/adamsIndustries-www/`
- **Hugo binary:** The repo vendors a macOS binary at `./bin/hugo` which won't run in the Linux sandbox. Download the Linux ARM64 build to `/tmp/hugo` if not already present.
- **Git remote:** `origin` on GitHub. A PAT is configured, so `git push` should work from the sandbox.

## Steps

### 1. Install Hugo in the sandbox (if needed)

Check if `/tmp/hugo` exists and is executable. If not, download:

```bash
curl -sL https://github.com/gohugoio/hugo/releases/download/v0.147.0/hugo_extended_0.147.0_linux-arm64.tar.gz | tar xz -C /tmp hugo
```

Verify with `/tmp/hugo version`. If the architecture is wrong (`Exec format error`), check `uname -m` and grab the matching build.

### 2. Build the site

```bash
cd /sessions/kind-dazzling-fermi/mnt/adamsIndustries-www/ && /tmp/hugo --minify 2>&1
```

Report the page count and any warnings. If the build fails, stop and troubleshoot — do not push broken output.

### 3. Auto-generate a commit message

Look at `git diff --stat` and `git status` output. Write a short, lowercase commit message summarizing what changed. Examples:

- `add turnstile site key and form endpoint to hugo.toml`
- `update homepage hero copy and fix mobile nav`
- `new blog post: survey deadline approaching`

Keep it under ~72 characters. No conventional-commit prefixes unless the user asks for them.

### 4. Stage, commit, and push

```bash
cd /sessions/kind-dazzling-fermi/mnt/adamsIndustries-www/ && \
  git add -A && \
  git status && \
  git commit -m "<auto-generated message>" && \
  git push origin main 2>&1
```

If git operations fail due to sandbox permissions (e.g., `.git/index.lock`, permission denied), fall back gracefully: show the user the exact commands to paste in their local terminal.

### 5. Report

Tell the user:
- Build result (pages, static files, any warnings)
- What was committed (short summary)
- Push result (success or fallback commands)

## Edge cases

- **Rebase in progress:** If `git status` shows a rebase, try `rm -f .git/index.lock && git rebase --abort` first. If that fails, tell the user to resolve it locally.
- **Nothing to commit:** If `git status` shows a clean tree, say so and skip the commit/push.
- **Build warnings:** Report them but still push unless there are actual errors.
